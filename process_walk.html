Walking NT Kernel Processes Like a White Man

March 11, 2022

Processes in the NT kernel are represented by an EPROCESS object. The EPROCESS inherits a KPROCESS. They can be viewed via public symbols. Regardless of what your goal might be, operating on the EPROCESS and KPROCESS object grants access to information like directory table base, image file name, process id, thread list, VAD tree, etc.

There are many ways to acquire an EPROCESS pointer by file name or process id. In this post I will cover some methods, their limitations, and what the more correct approach is.

The kernel exports PsInitialSystemProcess which holds a pointer to the EPROCESS instance of the system process. The EPROCESS struct holds a couple of fields that can be used to walk linked processes from the initial system process.

(1) ActiveProcessLinks
(2) MmProcessLinks
(3) SessionProcessLinks
(4) ProcessListEntry

In many ways, these links largely achieve the same goal, save SessionProcessLinks. We do not wish to discriminate processes by session. We walk any of the outstanding links. Each link is the field entry in a doubly linked list, meaning you do not get the pointer to the EPROCESS instance, but the field offset from it. This will have to be accounted for.

Walking the list is rather trivial. Finding the correct EPROCESS by image file name is less so. The EPROCESS struct holds a field ImageFileName for this purpose, but for reasons I cannot explain (((Microsoft))) decided this buffer should be 15 bytes long, making some images virtually indistinguishable. We solve this by getting the SeAuditProcessCreationInfo instead. The field is an SE_AUDIT_PROCESS_CREATION_INFO, which only holds a pointer to an OBJECT_NAME_INFORMATION. The OBJECT_NAME_INFORMATION struct holds a UNICODE_STRING. The UNICODE_STRING struct looks like this.

When a unicode string struct is initialised, the length is set equal to the byte length of the buffer, and the capacity to the byte length plus 2. I suppose this is to make space for the null terminator. Regardless, reading the unicode string from the process creation info gives us the DOS device path of the process image. From here we look for the desired image name within the unicode string buffer. If it is found, we return the EPROCESS instance.

[[ nodiscard ]]
const std::ptrdiff_t process_by_name(
   const std::wstring_t process
) {
   auto ctx{ nt::export( L"PsInitialSystemProcess" ) };
   if ( !ctx )
      return 0;

   static auto list{ nt::sym( L"EPROCESS", L"MmProcessLinks" ) };
   static auto name{ nt::sym( L"EPROCESS", L"SeAuditProcessCreationInfo" ) };

   while ( ctx = *ptr< std::ptrdiff_t* >( ctx + list ) - list, ctx ) {

      auto uni{ read< virt, unicode_t* >( ctx + name ) };
      if ( !uni
        || !uni->m_length
        || !uni->m_buffer )
         continue;

      if ( wcsstr( uni->m_buffer, process ) )
         return ctx;
   }
   return 0;
}
